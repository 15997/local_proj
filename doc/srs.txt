1. Introduction 
1.1. Purpose 
The primary objective of this Software Requirements Specification (SRS) document is to provide 
a comprehensive and detailed description of the functional and non-functional requirements for 
the Addis Local Service Finder system. This document serves as the foundational agreement 
between the development team, project advisors, and stakeholders regarding what the system 
will accomplish. 
After reading this SRS document, readers—including developers, testers, project managers, and 
academic evaluators—should have a clear and unambiguous understanding of: 
● The system's overall intent and the urban problem it aims to solve in Addis Ababa 
● The specific features, capabilities, and constraints of the proposed software. 
● The interactions between different user roles and the system.  
● The technical specifications necessary for design, implementation, and testing. 
The General Description section provides a high-level view of the product's context and user 
ecosystem. The Specific Requirements section details all functional requirements through 
structured tables and use cases, and defines critical non-functional attributes such as 
performance, security, and reliability. 
1.2. Scope 
The Addis Local Service Finder is a digital platform consisting of a Unified Mobile Application 
and a secure Admin Web Portal. The system's scope encompasses the following: 
● For Service Users (Clients): The system provides residents of Addis Ababa with a trusted 
directory to discover local service providers (e.g., plumbers, electricians, mechanics, 
tutors, guards). Users can search via: 
1. Traditional Browsing: Using filters for service type, location, rating, and 
availability. 
2. RAG AI Chatbot: A conversational interface allowing natural language queries 
(e.g., "Find a reliable electrician near Bole"). 
● For Service Providers: The system offers skilled tradespeople a platform to create 
verified digital profiles, manage their service listings, and connect with potential clients. 
● For Administrators: The system includes a secure web portal for managing the platform's 
integrity. Key functions include:

➢ Performing CRUD operations on all platform data. 
➢ Reviewing and vetting external information retrieved by the RAG system before

Out of Scope (Explicitly Excluded) 
1. Payment Processing: The system will not handle financial transactions. Payments are

3. Advanced Background Checks: Verification relies on manual document review by 
admins, not official criminal background checks. 
4. Unvetted Internet Data: The RAG system's external retrieval is a staged process; 
information from the public internet is not directly presented to users without prior admin 
verification.

2. General Description 
2.1. Product Perspective 
The Addis Local Service Finder is a self-contained software system designed to bridge a critical 
market gap in Addis Ababa's rapidly urbanizing environment. It operates as an independent 
platform but will interact with external mapping services (for geolocation) and potentially search 
engine APIs (for the enhanced RAG functionality). 
The Problem Context (Traditional & Digital Systems): 
● Traditional System (Word-of-Mouth): The dominant, inefficient method relies on 
fragmented personal networks. This leads to high search friction, inability to verify 
quality, and no accountability for service providers. 
● Existing Digital Listings: General online directories or social media groups lack 
verification, structured data, and intelligent search capabilities. Users face information 
overload with outdated or unreliable listings. 
What Addis Local Service Finder Offers: 
1. Centralization & Verification: It replaces fragmented information with a single, 
admin-verified directory, establishing trust. 
2. Intelligent Access: It surpasses basic keyword search with a RAG-powered AI Chatbot, 
allowing users to find services through natural conversation. 
3. Dual-Service Model: It categorically supports both short-term Task-Based Services (e.g., 
plumbing repair) and long-term Contract-Based Employment (e.g., security guard), 
addressing a wider range of urban needs. 
4. Managed Openness: Unlike a completely closed system, its enhanced RAG architecture 
can responsibly incorporate external public information through a strict admin 
verification gateway, ensuring data quality is maintained while expanding its knowledge 
base. 
The system is innovative, acting not just as a directory but as an intelligent conversational agent 
powered by verified local data.

2.2. Product Functions 
The core mission of the Addis Local Service Finder is to reduce search friction and build trust in 
finding local services. The product will perform the following primary functions: 
1. Facilitate Conversational Discovery: Provide a RAG AI Chatbot interface where users 
can ask complex, natural language questions about services and receive accurate, 
context-aware responses sourced from the verified database. 
2. Enable Filtered Browsing: Allow users to browse service providers using intuitive 
filters (service category, location on a map, customer rating, availability status). 
3. Manage a Verified Provider Ecosystem: 
● Allow service providers to register and create detailed professional profiles. 
● Provide administrators with tools to verify provider credentials and 
documentation before profile activation. 
● Implement a staging area to review and clean external data retrieved by the RAG 
system before integration. 
4. Connect Users and Providers: Enable direct communication between clients and 
verified providers through contact information displayed on profiles. 
5. Maintain Quality through Feedback: Implement a review and rating system where 
clients can rate services and provide public feedback, creating a reputation mechanism 
for providers. 
6. Provide Administrative Oversight: Offer a comprehensive web dashboard for system 
administrators to monitor key metrics (user counts, pending verifications), manage all 
data, and ensure platform integrity

2.4. General Constraints 
The development and operation of the Addis Local Service Finder system are subject to the 
following general constraints: 
1. Technological Constraints: 
● The mobile application must be built using a cross-platform framework (Flutter) 
to run on both Android (version 5.0+) and iOS (version 8.0+) devices. 
● The backend and RAG AI components will be implemented primarily using 
Python (with frameworks like FastAPI) 
● The system requires a Vector Database (e.g., Chroma DB) for efficient semantic 
search within the RAG pipeline. 
● External data retrieval for the enhanced RAG feature will depend on the 
availability and terms of third-party search engine APIs or web scraping 
protocols. 
2. Economic & Operational Constraints: 
● Initial development is an academic project with zero labor cost. Post-deployment 
operational costs (cloud hosting, domain, app store fees) are estimated at ~65,000 
ETB initially and ~5,500 ETB/month. 
● The system's data verification model requires human administrative labor, 
creating an ongoing operational cost and a potential bottleneck for scaling. 
3. Schedule Constraints: The entire project must be designed, implemented, tested, and 
documented within a two-semester academic timeline (approximately 6-8 months), as per 
the university's industrial project guidelines. 
4. Regulatory & Ethical Constraints: The system must comply with relevant Ethiopian 
data protection guidelines. The manual verification step for external data is a critical 
constraint to prevent the dissemination of unverified or harmful information.

3.4.2. Reliability 
1. Availability: The core service (mobile app backend and admin portal) shall have an 
uptime of 99.5% during business hours (6:00 AM - 10:00 PM local time). 
2. Data Integrity: The system shall ensure that all user data (profiles, reviews, transactions) 
is consistently saved and retrieved without corruption. Database transactions shall follow 
ACID properties. 
3. Error Recovery: In the event of a network interruption during a critical operation (e.g.,

maintenance shall not exceed 2 hours per month. 
3. Admin Portal: The Admin Web Portal shall be accessible to authorized personnel at all 
times from designated secure networks. 
3.4.4. Security 
1. Data in Transit: All communications between the mobile app, admin portal, and the 
server shall be encrypted using HTTPS (TLS 1.2 or higher). 
2. Authentication: User authentication shall be performed using secure, industry-standard 
methods. Passwords shall be hashed using a strong, salted hashing algorithm (e.g., 
bcrypt) before storage. Session management shall utilize short-lived JWT (JSON Web 
Tokens). 
3. Authorization: The system shall enforce role-based access control (RBAC): 
● Clients can only modify their own data and reviews. 
● Providers can only modify their own profiles and view their inquiries. 
● Admins have full CRUD privileges but their actions shall be logged for audit 
trails. 
4. Data Privacy: Personal Identifiable Information (PII) such as phone numbers and email 
addresses shall be masked or hidden from public view by default, only revealed when a 
user explicitly initiates contact (for providers) or is logged in as an admin.

1. Code Quality: The codebase shall follow consistent coding standards (PEP 8 for Python, 
Dart/Flutter conventions) and shall be documented with clear comments, especially for
