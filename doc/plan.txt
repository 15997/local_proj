# Addis Local Service Finder - Comprehensive Documentation and Implementation Analysis

## Overview

The Addis Local Service Finder is a comprehensive digital platform designed to connect residents of Addis Ababa with verified local service providers (plumbers, electricians, mechanics, tutors, security guards, etc.). The system features an innovative RAG (Retrieval-Augmented Generation) AI chatbot for conversational service discovery, along with traditional browsing capabilities, all built on a foundation of trust through administrative verification.

---

## Part 1: Documentation Analysis

### 1. Software Requirements Specification (SRS) - In Depth

The SRS document serves as the foundational agreement between stakeholders, defining WHAT the system must accomplish.

#### 1.1 Purpose and Scope

**Primary Objective:** The SRS establishes a comprehensive blueprint that ensures all stakeholders (developers, testers, project managers, academic evaluators) share an unambiguous understanding of the system's intent.

**Problem Context:**
- **Traditional System (Word-of-Mouth):** Fragmented, inefficient, lacks accountability
- **Existing Digital Listings:** Unverified, outdated, poor search capabilities
- **Solution Gap:** No trusted, intelligent directory exists for Addis Ababa

**System Scope:**
- **Mobile Application:** Unified app serving both clients and service providers
- **Admin Web Portal:** Secure management interface
- **Core Functions:** Conversational AI search, filtered browsing, provider verification, review system

**Explicitly Excluded (Out of Scope):**
- Payment processing (direct arrangements only)
- Dispute resolution
- Criminal background checks
- Unverified internet data exposure

#### 1.2 User Roles and Characteristics

| Role | Characteristics | Permissions |
|------|----------------|-------------|
| **Client/Service Finder** | Addis Ababa residents seeking services | Search (chatbot & filters), save providers, submit reviews, contact providers |
| **Service Provider** | Skilled professionals offering services | Create profiles, update availability, upload documents, respond to inquiries |
| **System Administrator** | Trusted staff ensuring platform integrity | Verify providers, review external data, full CRUD operations, monitor analytics |

#### 1.3 Functional Requirements (FR-01 to FR-10)

**FR-01: User Registration & Role Selection**
- Users choose between Client or Service Provider roles
- Immediate activation for clients; providers enter pending verification
- Input validation for email/phone format and password strength

**FR-02: User Login & Authentication**
- JWT-based authentication
- Role-specific dashboard loading
- Provider accounts restricted until verification complete

**FR-03: Traditional Service Browsing & Filtering**
- Filters: Service category, location, rating, availability, price range
- Sort options: Rating, distance, relevance
- Unregistered users: View-only mode

**FR-04: RAG AI Chatbot Conversational Search**
- Natural language query processing
- Vector database semantic search
- Response generation with verified data only
- Confidence scoring with external fallback

**FR-05: Service Provider Profile Management**
- Business details, service categories, pricing
- Availability status toggle
- Document upload for verification

**FR-06: Service Provider Verification (Admin)**
- Document review workflow
- Approve/Request Info/Reject options
- Verified status required for public visibility

**FR-07: External Data Retrieval & Staging**
- Retrieves public service information
- Staging database for unverified data
- Admin review before integration

**FR-08: Review & Rating System**
- 1-5 star ratings with optional comments
- Requires verified interaction
- Admin moderation for inappropriate content

**FR-09: Geolocation-Based Search**
- GPS or manual location selection
- Proximity-based ranking
- Radius filtering

**FR-10: Admin Dashboard & Analytics**
- Real-time metrics (users, verifications, activity)
- Exportable reports
- System health monitoring

#### 1.4 Non-Functional Requirements

**Performance:**
- Search results: ≤3 seconds for 95% of queries
- RAG response: ≤5 seconds including retrieval and generation
- Mobile cold start: ≤2 seconds on mid-range devices

**Reliability:**
- Uptime: 99.5% during business hours (6 AM - 10 PM)
- ACID compliance for transactions
- Graceful degradation during network interruptions

**Security:**
- HTTPS with TLS 1.2+ for all communications
- bcrypt password hashing
- JWT with short-lived tokens
- Role-Based Access Control (RBAC)
- Input validation/sanitization against injection attacks

**Maintainability:**
- Modular architecture
- Comprehensive logging (90-day retention)

**Portability:**
- Flutter for cross-platform mobile (Android 5.0+, iOS 8.0+)
- React admin portal (latest Chrome, Firefox, Edge)

#### 4.2 RAG System Implementation

**Components:**

**A. Vector Database (ChromaDB)**
```python
import chromadb
from chromadb.utils import embedding_functions

class VectorRetriever:
    def __init__(self):
        self.client = chromadb.PersistentClient(path="./chroma_db")
        self.embedding_fn = embedding_functions.SentenceTransformerEmbeddingFunction(
            model_name="all-MiniLM-L6-v2"
        )
        self.collection = self.client.get_or_create_collection(
            name="service_providers",
            embedding_function=self.embedding_fn
        )
    
    def add_provider_embeddings(self, provider_id, text):
        """Add provider profile to vector DB"""
        self.collection.add(
            documents=[text],
            metadatas=[{"provider_id": provider_id}],
            ids=[f"provider_{provider_id}"]
        )
    
    def semantic_search(self, query, top_k=10):
        """Search for semantically similar providers"""
        results = self.collection.query(
            query_texts=[query],
            n_results=top_k
        )
        return results
```

**B. LLM Integration (LangChain)**
```python
from langchain import LLMChain
from langchain.llms import HuggingFacePipeline
from langchain.prompts import PromptTemplate

class RAGService:
    def __init__(self, db_session):
        self.db = db_session
        self.vector_retriever = VectorRetriever()
        self.llm = HuggingFacePipeline.from_model_id(
            model_id="meta-llama/Llama-2-7b-chat-hf",
            task="text-generation"
        )
        
    async def process_query(self, query: str, user_id: int, location: dict):
        # Step 1: Parse query intent
        intent = self.parse_intent(query)
        
        # Step 2: Generate embedding and search vector DB
        relevant_providers = self.vector_retriever.semantic_search(query)
        
        # Step 3: Evaluate confidence
        confidence = self.evaluate_confidence(relevant_providers)
        
        if confidence < CONFIDENCE_THRESHOLD:
            # Trigger external search
            return await self.handle_low_confidence(query)
        
        # Step 4: Generate response with LLM
        response = self.generate_rag_response(query, relevant_providers)
        
        return RAGResponse(
            answer=response,
            sources=relevant_providers,
            data_source="verified_database",
            confidence_score=confidence
        )
```

#### 4.3 External Data Pipeline

```python
class ExternalDataRetrievalComponent:
    def __init__(self):
        self.search_apis = [GoogleSearchAPI(), BingSearchAPI()]
        self.web_scraper = ControlledWebScraper(
            delay=2,  # Politeness delay
            allowed_domains=['*.ethiopia.com', '*.addis.com']
        )
        self.staging_manager = StagingManager()
    
    async def retrieve_external_providers(self, query: str):
        """Retrieve provider data from external sources"""
        results = []
        
        # Try search APIs first
        for api in self.search_apis:
            try:
                api_results = await api.search(f"{query} service provider Addis Ababa")
                results.extend(api_results)
            except Exception as e:
                log_error(f"API search failed: {e}")
        
        # Stage results for verification
        staged_ids = []
        for result in results:
            # Validate data quality
            quality_score = self.calculate_quality_score(result)
            
            if quality_score >= QUALITY_THRESHOLD:
                # Check duplicates
                duplicates = self.check_duplicates(result)
                
                staging_id = await self.staging_manager.stage_data(
                    data=result,
                    source_url=result.get('url'),
                    quality_score=quality_score,
                    potential_duplicates=duplicates
                )
                staged_ids.append(staging_id)
        
        # Notify admin
        if staged_ids:
            await notify_admin_new_staging(staged_ids)
        
        return staged_ids
```

#### 4.4 Verification Workflow

```python
class AdminVerificationComponent:
    def __init__(self):
        self.staging_repo = StagingRepository()
        self.provider_repo = ProviderRepository()
        self.audit_logger = AuditLogger()
    
    async def approve_staged_data(self, staging_id: str, admin_id: str, edits: dict = None):
        """Approve and integrate staged external data"""
        
        # Retrieve staged data
        staged = await self.staging_repo.get_by_id(staging_id)
        
        # Apply edits if provided
        if edits:
            staged.data.update(edits)
        
        # Check if this matches an existing provider
        if staged.potential_duplicates:
            # Merge with existing provider
            provider = await self.merge_with_existing(staged)
        else:
            # Create new provider
            provider = await self.create_new_provider(staged.data)
        
        # Update staging record
        await self.staging_repo.update_status(
            staging_id, 
            "APPROVED",
            reviewed_by=admin_id
        )
        
        # Log action
        await self.audit_logger.log(
            admin_id=admin_id,
            action="APPROVE_EXTERNAL_DATA",
            target=staging_id,
            details={"merged": bool(staged.potential_duplicates)}
        )
        
        # Notify provider (if contact info available)
        await self.notify_provider_creation(provider)
        
        return provider
```

### 5. Integration and Data Flow

#### 5.1 User Registration Flow
1. User submits registration form (mobile app)
2. Frontend validates input format
3. API endpoint receives request
4. Backend checks for existing email/phone
5. Password hashed with bcrypt
6. User record created in PostgreSQL
7. Role-specific profile created
8. JWT token generated and returned
9. Mobile app stores token and navigates to role-specific home

#### 5.2 RAG Search Flow
1. User enters natural language query
2. Mobile app sends query to `/api/v1/rag/query`
3. Backend RAG service:
   - Parses query intent
   - Generates embedding vector
   - Searches ChromaDB for similar provider profiles
   - Retrieves top K results
   - Calculates confidence score
4. If confidence ≥ threshold:
   - Passes results to LLM for response generation
   - Returns response with verified sources
5. If confidence < threshold:
   - Triggers external search pipeline
   - Data staged for admin review
   - Returns fallback response
6. Interaction logged to `rag_interactions` table

#### 5.3 External Data Processing Flow
1. Scheduled job or on-demand trigger
2. ExternalDataRetrievalComponent queries search APIs
3. Results parsed and structured
4. Quality score calculated
5. Duplicate check against existing providers
6. Data inserted into `verification_staging` table
7. Admin notified via dashboard
8. Admin reviews and approves/rejects
9. On approval:
   - Data moved to `service_provider_profiles`
   - Embeddings generated and stored in ChromaDB
   - Staging record updated

```python
# JWT token generation
def create_access_token(data: dict, expires_delta: timedelta = None):
    to_encode = data.copy()
    expire = datetime.utcnow() + (expires_delta or timedelta(minutes=15))
    to_encode.update({"exp": expire})
    encoded_jwt = jwt.encode(to_encode, SECRET_KEY, algorithm=ALGORITHM)
    return encoded_jwt

# Password hashing
pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")

def verify_password(plain_password, hashed_password):
    return pwd_context.verify(plain_password, hashed_password)

def get_password_hash(password):
    return pwd_context.hash(password)

# Role-based access control
def require_role(required_role: str):
    def decorator(func):
        @wraps(func)
        async def wrapper(*args, **kwargs):
            current_user = kwargs.get('current_user')
            if required_role not in current_user.roles:
                raise HTTPException(status_code=403, detail="Insufficient permissions")
            return await func(*args, **kwargs)
        return wrapper
    return decorator
```

### 10. Key Innovations and Differentiators

1. **RAG-Powered Conversational Search**
   - Natural language understanding
   - Semantic search over verified providers
   - Confidence-based fallback to external sources

2. **Human-in-the-Loop Verification**
   - All external data requires admin review
   - Staging database isolates unverified information
   - Duplicate detection and merge capabilities

3. **Dual-Service Model**
   - Task-based services (plumbing, electrical repairs)
   - Contract-based employment (guards, tutors)
   - Different interaction patterns for each type

4. **Hybrid Database Architecture**
   - PostgreSQL for structured data and ACID compliance
