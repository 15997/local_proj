1. Introduction 
1.1 Purpose 
This Software Design Specification (SDS) document provides a comprehensive architectural and 
design blueprint for the Addis Local Service Finder system. The document serves as a technical 
guide for the development team, detailing the system's architecture, component designs, 
algorithms, and implementation strategies. It translates the requirements specified in the 
Software Requirements Specification (SRS) into a concrete design that ensures the system's 
reliability, scalability, security, and maintainability. 
The SDS documents the system's decomposition into subsystems, defines the object model with 
detailed class specifications, outlines critical algorithms (particularly for the RAG system), 
specifies API interfaces, details security implementations, defines database schemas, and 
describes deployment strategies. This document will guide all implementation phases and serve 
as a reference for testing, maintenance, and future enhancements. 
1.2 General Overview 
The Addis Local Service Finder is an intelligent digital platform designed to address the 
challenge of finding reliable local service providers in Addis Ababa. The system combines 
modern web technologies with advanced AI capabilities to create a trusted, efficient service 
discovery ecosystem. 
Core System Components 
Unified Mobile Application: A cross-platform application built with Flutter that serves both 
service seekers (clients) and service providers, with dynamic interface adaptation based on user 
role. 
RAG-Powered AI Chatbot: The system's intelligence core that enables natural language 
conversational search for services. The RAG (Retrieval-Augmented Generation) system retrieves 
information from a verified local database and can extend to external web sources with 
administrative oversight. 
Admin Web Portal: A secure management interface for system administrators to verify service 
providers, review external data, monitor system metrics, and maintain data integrity. 
External Data Pipeline: A controlled mechanism for retrieving public service information from 
the internet, staging it for verification, and integrating it into the trusted database after 
administrative approval. 
Multi-Database Architecture: A hybrid database system comprising:

1.3 Development Methods & Contingencies 
Development Methodology 
The project follows an Agile Software Development approach using the Scrum framework, 
adapted to accommodate the academic timeline and the complexity of RAG AI integration. 
Key Agile Practices 
1. Iterative Development: Two-week sprints with defined deliverables 
2. Incremental Delivery: Functional increments after each sprint 
3. Continuous Feedback: Regular reviews with advisor and peer testing 
4. Adaptive Planning: Flexible response to technical challenges 
Sprint Structure 
● Sprint 0: Environment setup and architectural foundation 
● Sprint 1–3: Core database and admin CRUD operations 
● Sprint 4–6: Mobile app UI/UX and basic browsing 
● Sprint 7–9: RAG AI Chatbot integration 
● Sprint 10–12: Advanced features and optimization 
Technical Stack 
● Frontend (Mobile): Flutter (Dart SDK), MVVM architecture, Provider pattern 
● Backend: Python 3.9+, FastAPI, LangChain, Hugging Face Transformers 
● Databases: PostgreSQL (primary), ChromaDB (vector), PostgreSQL (staging) 
● Admin Portal: React.js with Material-UI 
Contingency Planning 
Technical contingencies address risks including RAG system complexity, external data quality 
issues, geolocation accuracy, database performance, and admin verification bottlenecks. 
Mitigation strategies include fallback mechanisms, hybrid approaches, optimization techniques, 
and tiered verification systems. Project management contingencies address timeline risks, team 
resource distribution, and technical debt through parallel development, knowledge sharing, and 
dedicated refactoring sprints.

2.3 Access Control 
The Addis Local Service Finder system employs a Role-Based Access Control (RBAC) 
mechanism to ensure secure and controlled access to system resources. Access permissions are 
assigned according to predefined user roles, and each role is granted only the privileges 
necessary to perform its authorized operations. The RBAC model enhances system security, 
minimizes unauthorized actions, and ensures clear separation of responsibilities across users. 
2.3.1 User Roles 
a) Client (Service Seeker) 
Clients represent end-users who search for and interact with service providers. Their permitted 
actions include: 
● Searching and browsing available services 
● Viewing verified provider profiles 
● Saving providers and search preferences 
● Submitting ratings and reviews 
● Contacting service providers through the platform 
b) Service Provider 
Service Providers are users who offer professional services on the platform. In addition to all 
client capabilities, they are authorized to: 
● Create and manage service listings 
● Update service and availability status 
● Upload verification and supporting documents 
● Respond to client inquiries 
● View basic performance and engagement metrics

c) System Administrator 
Administrators are responsible for governance, verification, and platform supervision. Their 
privileges include: 
● All provider permissions 
● Verifying and approving service provider accounts 
● Reviewing and approving externally sourced provider data 
● Managing user accounts and role assignments 
● Monitoring platform activity and system metrics 
● Configuring system settings and operational policies 
● Accessing audit trails and system logs 
2.3.2 Authentication and Authorization Mechanism 
User authentication and authorization are enforced through a token-based authentication 
framework: 
● Authentication Method: JSON Web Token (JWT)–based authentication using the 
Bearer token scheme 
● Token Model: Short-lived access tokens and long-lived refresh tokens 
● Security Controls: HTTP-only storage for tokens 
● Password Policy: Minimum eight-character passwords with complexity requirements; all 
credentials are securely hashed before storage

3.1.2 Domain Overview 
The system is built upon a multi-role architecture designed to facilitate service discovery, 
interaction, and verification. It integrates a RAG (Retrieval-Augmented Generation) system for 
conversational queries and a structured review system tied to verified interactions. 
Core Domains and Relationships 
User and Authentication Domain 
● User (Concrete Class): Represents a system account with authentication credentials. 
● Roles: A User can possess multiple roles (Client, ServiceProvider, Admin) simultaneously, 
managed via a collection. 
● Associations: A User maintains a relationship with both ClientProfile and 
ServiceProviderProfile, which are instantiated based on assigned roles. 
Profile Domain 
● ClientProfile: Extends the User account with client-centric data such as saved providers 
and search history. 
● ServiceProviderProfile: Extends the User account with business details, service catalogs, 
and verification status. 
● Mandatory Link: Both profile types maintain a strict association with their parent User 
account. 
Service Interaction Domain

ServiceProvider. 
● Cardinality: A Client (via User role) can initiate many ServiceInquiry objects. 
A ServiceProvider (via User role) can receive many ServiceInquiry objects. 
● Review Integrity: A Review is strictly associated with a completed ServiceInquiry to

Service Discovery and RAG Domain 
● Service Offerings: A ServiceProviderProfile maintains a many-to-many relationship with

Verification and Management Domain

internal results falls below a predefined threshold. This is an on-demand process triggered by 
user queries rather than a scheduled batch job. 
Message Sequence 
Phase 1: Fallback Triggering 
● RAGQueryComponent evaluates internal results. 
● If Confidence < Threshold, RAGQueryComponent calls triggerExternalSearch(query, location) on 
the ExternalDataService. 
Phase 2: External Data Retrieval & Parsing 
● ExternalDataService sends scrapeWebSources(searchQuery) to the Web Scraper. 
● Web Scraper executes HTTP requests to External Websites (respecting rate limits). 
● External Websites return HTML/JSON responses to the Web Scraper. 
● Web Scraper passes raw data to the Data Parser. 
● Data Parser returns structured provider data to the ExternalDataService. 
● ExternalDataService calls validateDataQuality(data) on the Quality Validator. 
● Quality Validator returns a quality score and validation result. 
Phase 3: Data Staging 
● ExternalDataService calls insertStagedData(data) into the Staging Database. 
● Staging Database returns a stagingId. 
● ExternalDataService sends notifyNewStaging(stagingId) to the Admin Portal. 
Phase 4: Admin Notification 
● Admin Portal displays the notification to the Administrator via the verification queue. 
Architectural Design Decisions 
● Conditional Execution: The flow is strictly gated by the RAG system's confidence 
evaluation (linked to Figure 12 State Chart). 
● Verification First: External data is routed to a staging database and admin queue rather than 
direct production injection, ensuring the system prioritizes verified information.

4.8 Component Integration Specifications 
Inter-component Communication Protocols: 
1. Mobile App ↔ Backend: RESTful HTTP/HTTPS with JSON payloads 
2. Backend ↔ RAG Engine: gRPC for low-latency communication 
3. Backend ↔ Vector DB: ChromaDB client library over HTTP 
4. Backend ↔ External APIs: HTTPS with exponential backoff retry 
5. Admin Portal ↔ Backend: WebSocket for real-time notifications 
6. Database Replication: PostgreSQL streaming replication
